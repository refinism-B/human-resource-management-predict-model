# 直播錄影人力預測系統
這是一個ML專案，功能在於根據專案需求，預測需要安排的人力。

## 簡介
- 在直播錄影行業，**人力的需求非常彈性**，根據客戶的需求，可能需要一個8-10人的完整團隊，也可能只需要2-3人就能執行。
- 然而這樣的彈性就需要花費更多的時間和精力進行人力安排，**一方面要具備足夠的專業知識來了解每個需求對應的人力；另一方面也要對每個專案的需求都非常明確**，才能做到精準的人力安排，不多也不少。
- 在旺季時，一天可能就有2-3個案子，一週可能有5-6天都是滿班，除了處理當週的案子，還得為下週或未來幾週的案子做預備。這對於人力安排的工作造成了極大的壓力和困難。**而這便是這個專案嘗試解決的痛點**。

## 作法
- 在過去一年半中，收集了幾乎每一個拍攝專案的需求以及執行的人力、人數。
- 將這些資料輸入ML模型中進行訓練，並期望能根據這份資料學習到規律。
- 未來只要將專案需求輸入模型，便能預測所需的人力，最後只需要進行人為確認、聯絡發案給對應的工作人員即可。

## 資料集
1. 起初是透過**Google Drive雲端硬碟**記錄，在每次專案結束後，請工作同仁主動填寫。
2. 最初的目的僅只是工作紀錄，並給主管作為考績的參考資料，**並非以ML為目的蒐集**。
3. 基於第2點，所以**原始表格並不適合直接用於訓練模型**。在專案目錄中的Data目錄底下的檔案，是經過清理轉換過後的資料格式，已是較為正規的結構化資料。
4. 資料本身共有**21個欄位，469筆資料，時間跨度自2024年1月開始至2025年5月底**，大部分皆為數值欄位，有部分欄位採用**one-hot code**的方式來表明狀態（通常是0表示無、1表示有）。

## ML模型
- 選擇採用較為基本、容易訓練的**Random Forest迴歸模型**，並將其中的**20%資料設為測試集**，其餘皆用於訓練。
- 最終選擇的模型版本**mse為0.25705954094974826，rmse為0.5070103953073825**。

## 專案用法
- 程式已經寫好簡單的UI介面，只需要**執行跟目錄的main.bat檔案**，便會執行主程式。使用者只要依序輸入專案的需求，送出後便會顯示模型回傳的結果。
- 輸入資料的方式可以**手動單筆輸入**，也能**透過檔案將多筆資料一次輸入**。

## 未來展望
1. **資料特徵**：
    - **痛點**：直播攝影的專案需求其實切分非常細緻，**目前的特徵數量並未涵蓋所有可能的專案需求**。但在最初收集資料時便沒有將所有欄位納入，現在也已無法確認。
    - **展望**：未來若確認資料將用於ML訓練，在最初制定資料格式時，就應**盡量納入所有可能的特徵**。
2. **人員預測**：
    - **痛點**：目前模型**僅預測「人數」並未包含人員**，對人力安排工作者來說，雖然知道所需的人數，但**還是需要費心安排適當的人選**。
    - **說明**：原始資料中其實有紀錄每個專案執行的人員是誰，但由於**人員會異動（特別是員工到職或離職）**可能會導致模型預測失準，未來也可能發生人員異動但模型無法應付這樣的變動。故在訓練時先將人名資料移除，僅使用人數做訓練。
    - **展望**：為解決這個問題，預想還需要處理幾個要件：
        1. 建立**人力資料庫**：紀錄現有人力資源以及擅長項目（可以分數表示擅長程度）。
        2. 將人員安排加回訓練資料中呈現，讓模型不只知道每個專案執行的人數，也能知道**是由誰執行**。
        4. 然而模型學習的應該是從人力池中，**如何挑選適合的人選（可以擅長程度為人力資料的特徵）**，而不是死記人名或編號。
        3. 若人員有異動，**可隨時更新人力資料庫，模型也只能從當下的人力資料庫進行預測安排**。
